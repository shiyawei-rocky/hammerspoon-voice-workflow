# Hammerspoon 语音助手：指导思想与操作准则

## 1. 项目定位与目标
这是一个面向个人的系统级语音输入与文本处理工具，核心目标是“更快、更稳、更少手工步骤”。架构以 Hammerspoon 为胶水层，串联录音、云端 ASR 与 LLM 文本处理，在任何应用中都能完成“语音→文本→润色/翻译/结构化→粘贴”的闭环。

目标不是做成通用产品，而是保持主链路稳定、可控、可快速修复。

## 2. 核心原则（必须遵守）
- 稳定优先：任何改动必须先保证主链路可用（F5/F6/F7）。
- 不动主链路：除非出现明确故障，否则不改变“录音→ASR→LLM→输出”的核心流程。
- 避免状态机重构：当前系统已可用，重构只做低风险渐进式抽取。
- 降低惊扰：避免引入新的弹窗与阻塞流程，错误提示要明确、可恢复。
- 数据可回溯：输出可选记录到本地日志，便于复盘和词库增量更新。

## 3. 主链路与关键流程概览
- F5（语音录入）：录音 → 云端 ASR → LLM 后处理（f5_asr_post）→ 可选自动 F7 润色 → 粘贴输出。
- F6（选区 + 指令）：选区文本 + 语音指令 → LLM 执行（f6_selection）→ 可选自动 F7 润色 → 粘贴输出。
- F7（润色）：对选区进行轻度润色与校对（f7_polish）。
- Alt+E/W/Q/A/C/Z/X（转换类）：翻译/深度表达/结构化/摘要/纠错/步骤拆解/批判性分析。
- F8（一键助手）：按配置运行 digest 或 workflow，总结近期记录并产出项目情报。

提示词统一存放于 `prompts/prompts.json`，术语表在 `prompts/glossary.txt`，行为开关与路径在 `config.lua`。

## 4. 日常使用与操作习惯
- 使用前确保光标在可编辑文本区域，避免密码框或“安全输入”场景。
- 有选区时会优先处理选区，无选区则尝试全文捕获（可能受剪贴板与安全输入限制）。
- 长录音建议保持环境安静，避免中途频繁切换应用。
- 若输出“无明显变化”，多为原文过短或模型判断无需改动，可加长输入或调整提示词。
- 需要应用风格时，确保应用识别正确（App style 已在 `config.lua` 中配置）。

## 5. 维护与改动规则
- **允许的低风险改动**：
  - 修改 `prompts/prompts.json`（文案与风格控制）。
  - 调整 `config.lua`（模型、时序、功能开关、场景风格、日志与词库策略）。
  - 维护术语表 `prompts/glossary.txt`。
- **谨慎改动**：
  - 录音与 ASR 脚本（`whisper_cloud_transcribe.sh`、`llm_stream.sh`）。
  - 流式输出链路与轮询逻辑。
- **不建议做**：
  - 引入复杂状态机或大规模重构热键逻辑。
  - 改动主链路中的核心回调顺序（容易导致“错乱输出”）。
- 每次修改后，至少验证 F5/F6/F7/Alt+E/Alt+W/F8 的基本流程。

## 6. 风险与边界
- **剪贴板依赖**：全文捕获依赖复制/粘贴，用户在等待期间复制会污染结果。
- **安全输入限制**：安全输入开启时无法捕获文本。
- **网络与延迟**：云端 ASR 与 LLM 受网络波动影响，可能超时或失败。
- **外部依赖**：`ffmpeg`、`jq` 等工具缺失会导致脚本失败。
- **日志与外发**：转录日志与 flomo 同步默认开启时，要注意隐私与数据范围。

## 7. 演进路线（低风险优先）
- 首选：常量化与服务层抽取（已完成）+ 少量清理与一致性修复。
- 其次：统一配置来源，减少流式/非流式分叉。
- 可选：对“全文捕获失败率高”的场景做轻量补偿（重试、提示、回退策略）。
- 不建议：在未明确收益前进入状态机重构。
